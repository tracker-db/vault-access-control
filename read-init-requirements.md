## Module pinning
Always reference the module repo pinned to a commit SHA in source blocks:
```hcl
source = "git::https://github.com/tracker-db/modules-terraform-vault-rbac.git//modules/vault-rbac?ref=<COMMIT_SHA>"
```
Never use a branch name as `ref` in production — branches move.

   - `export VAULT_ADDR="https://127.0.0.1:8200"`
   - `export VAULT_TOKEN="s.yourtoken"`  # Prefer VAULT_TOKEN already set in your shell

## Naming convention
Use kebab-case and env prefix:
- `{env}-{service}-{sa|role|policy}-{purpose}`
- Examples:
  - `dev-myapp-sa`
  - `prod-billing-role-readonly`
  - `dev-myapp-read.hcl`

## Secrets handling
- Do not store `VAULT_TOKEN` or secrets in repo.
- Use `VAULT_TOKEN`/`VAULT_ADDR` or Vault agent; document alternative auth flows in `tasks.md`.

## Minimal test plan (commands + expected)
1. Verify Vault auth and connectivity:
   - `vault status`
     - expected: initialized, sealed: false, version: 1.18.2
2. Terraform plan shows intended changes:
   - `terraform plan -var="environment=dev"`
     - expected: plan shows creation of policies, roles, and service accounts as defined.
3. Apply and validate policy effect:
   - `terraform apply -var="environment=dev" -auto-approve`
   - Use Vault CLI to test token/identity permissions (example):
     - `vault token create -policy="dev-myapp-read" -period="1h" -format=json | jq -r .auth.client_token`
     - `VAULT_TOKEN=<that token> vault kv get secret/data/myapp/config`
     - expected: read succeeds for permitted path, denied for others.

## Rollback/backout
- To rollback a recent apply:
  - Use git to revert the change in branch `phase1`, re-run `terraform plan/apply` against dev.
- For immediate revocation:
  - `vault token revoke -accessor <accessor_id>`
  - `vault write sys/policies/acl/dev-myapp-read policy=""`  # to remove policy
